# StackShare benchmark and content requirements

## Required five-peer method

1. Browse the intended category and state why it matches the product’s primary developer job.
2. Select exactly five live, comparable tools using a visible category, relevance, and current profile completeness.
3. Browse each StackShare profile, its official product site, and at least one live alternatives/comparison page.
4. Capture evidence, not copied prose. Do not use only search snippets or a ranking from a third party.
5. Distinguish platform-controlled/profile-controlled fields from community-generated signals.

## Content plan

Write explicit buyer questions: What does it help developers do? Where does it run? How does it connect? What is the
implementation path? What is the licensing/pricing model? Which established tools might replace or complement it?
Use evidence-backed answers, short scanning sections, and technical specificity. Avoid generic claims such as
“best,” “leading,” “simple,” “secure,” or “scalable” unless a source explains the relevant mechanism and scope.

## Profile-controlled versus community-generated

| Surface | Treatment |
|---|---|
| Name, vendor, site/CTA, overview, category, features | Verify and prepare source-backed copy |
| Integrations, alternatives/comparisons | Include only genuine, documented relationships |
| Logo/screenshots/video | Use approved current assets; verify rights and secrets |
| Stacks, follows, votes, rank | Observe/measure; never promise, buy, or solicit deceptively |
| Pros/cons, adoption, discussion/decision records | Community content; do not author it as an independent user |
