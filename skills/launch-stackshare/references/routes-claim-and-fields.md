# StackShare routes, claim, and field capture

## Listing model

StackShare presents developer tools and company tech stacks. Start by searching for an existing tool or company
profile. A product vendor normally needs a tool profile; an engineering organization may also maintain a company
stack only when each disclosed technology is actually used.

Public tool profiles observed July 23, 2026 include a product title, vendor/byline when supplied, official website
or Try It CTA, category and rank where available, overview, key features, pros/cons, alternatives/comparisons,
adoption, discussions, and integrations. Counts for stacks, follows, votes, decisions, and pros/cons are community
signals: record them as observed, not as profile copy or a promised outcome.

## Verified ownership

StackShare’s current “Getting Verified” documentation says the verified badge appears once a tool is successfully
claimed. Requirements and routes:

| Method | Conditions | Steps |
|---|---|---|
| Domain email | Official website URL is correct; owner can receive email at the tool domain | Claim this tool → enter domain email → use inbox verification link |
| GitHub | Tool is open source and the signer has admin or maintain permissions on linked repository | Claim this tool → connect GitHub → permissions are verified |
| Support | Neither standard method works | Use current support path and proof of official role |

Verification is not permission to make unproven claims. Keep identity and information current; documentation says
verification may be revoked for ownership changes, guideline violations, or materially stale/misleading information.

## Live form protocol

The exact create/edit UI can change. In a real browser, record every visible field, help text, required mark,
character/file constraint, select value, upload requirement, and post-submit state. Do not infer that a public page
surface is an editable form field. Do not submit without approval.
