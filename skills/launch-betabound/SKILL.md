# Launch on BetaBound

## Goal

Create `.ulpi/launch/betabound/` with `ELIGIBILITY.md`, `BENCHMARK.md`, `BETA-LANDING-PAGE-PACKET.md`,
`ANNOUNCEMENT-PACKET.md`, `TESTER-SCREENER.md`, `FORM-CAPTURE.md`, `FEEDBACK-OPERATIONS.md`, `analytics.md`, and
positioning.

## Workflow

### 1. Prove beta eligibility

Return `BETA ANNOUNCEMENT READY`, `LANDING PAGE FIRST`, `PRODUCT NOT ELIGIBLE`, or `FIELD RECAPTURE NEEDED`. Confirm
the actual beta/unreleased feature, free access, working beta landing page, applicant path, clear feedback method,
responsible test owner, support capacity, privacy notice, expected timeframe, and valid expiry date. Do not reframe a
released product as a beta merely to qualify.

### 2. Benchmark exactly five live opportunities

Use `$browse` to inspect the current appropriate beta type and exactly five comparable opportunities. Capture the
title, concise description, public access/test type, audience, requirements, incentive framing, current/approaching
status, expiry, and application CTA. Build differentiated, evidence-backed copy and select only factual requirements.

### 3. Build the beta and feedback packet

Prepare the company/contact owner; product name; clear one- or two-sentence summary; true product type and audience;
access model; timeframe; expiration; tester target; technical, geographic, device, demographic, and accessibility
requirements; incentive terms; test application URL; product-information URL; and additional context. The beta landing
page must explain what is being tested, eligibility, time commitment, how to apply, how testers give feedback, data
handling, support, incentive terms, and end-of-test handling. Use a qualifying survey where location or capability
matters; BetaBound does not supply tester emails or collect feedback for the company.

### 4. Capture and review the live form

Map every current form field to an approved source, including the required controls listed above. Verify the beta URL
goes to an actual test-application page rather than a generic trial. Check one-line copy against live peers, prevent
unsupported performance/security claims, and write a source register. Present the full announcement, application
flow, incentive language, data/privacy handling, and expiry for approval.

### 5. Submit and operate a respectful test

After explicit approval, submit once and save the confirmation. If BetaBound sends a draft, verify its facts, CTA,
requirements, incentive, test status, dates, and URL before approving. Operate selection, onboarding, feedback,
support, consent, opt-out, and tester data handling independently. Track qualified applications, selection rate,
activation, completion, feedback themes, defect severity, retention, and incentive cost—not promised traffic. Request
edits/removal through the current BetaBound contact route when the test changes or closes. End `LIVE — BETABOUND TEST
RUNBOOK ACTIVE`.
