## Browse research protocol

Use one `$browse` session:

1. Open AlternativeTo and wait for network idle.
2. Search the exact product, creator, domain, prior names, and spelling variants.
3. Search the closest known competitor and task/type terms.
4. Inspect the visible sort and platform/cost filters.
5. Open exactly five products with the same primary job.
6. Open vendor-owned sites to verify current claims.
7. Record concise facts, dates, and URLs; never copy full profile or review text.

If ordinary Chromium receives an access block, do not switch to stealth or bypass controls without an approved
handoff. Use official indexed AlternativeTo pages and user-supplied screenshots, and mark anything not live-
verified `BLOCKED` or `PROVISIONAL`.

## Top-five selection

Select products a real user would expect when asking for alternatives to the best-known product for the same
main job. Record:

- search/profile URL and visible order
- active platform, cost, license, and other filters
- main task/focus and target user
- short/full description structure
- cost/license and platforms
- application types, tags, and highlighted features
- official/store/source/social links
- icon, screenshot, and video strategy
- number/quality of alternative associations
- likes, comments/reviews, and last-update signals as context only

Do not label products category leaders without evidence. AlternativeTo rank uses likes and undisclosed other
signals; a high like count alone is not the ranking formula.

## Main task and focus

Define one sentence:

`[Product] primarily helps [user] perform [main task] through [core focus/mechanism].`

Use this for taxonomy and alternatives. Secondary capabilities do not redefine the primary job. A large suite
can be an alternative only where the job a searcher wants matches the suite's central focus, not an incidental
feature.

## Description strategy

Draft three options:

- Task-led: `[Product] is a [type] that helps [user] [primary task].`
- Differentiation-led: name the main task, then a verified distinctive mechanism or delivery model.
- Migration-led: explain the same job and the meaningful platform, cost, hosting, or workflow difference.

Short Description should be one concise sentence naming the main purpose. Full Description should explain:

- primary user and task
- current core capabilities
- supported workflow/platforms
- meaningful differentiators
- relevant pricing/edition or hosting context

Do not include URLs, email, phone, physical addresses, calls for likes, rank claims, or unverifiable hype.
AlternativeTo permits official-material copy, but write a clean product-specific version to avoid stale or
contextless marketing language.

## Tags, application types, and features

- Application Type: subcategory/main purpose. Choose the closest exact live type.
- Tags: generic concepts/categories describing what the app is.
- Features: concrete options or support the app actually provides.

For each proposed term record exact live label, type, evidence, plan limitations, and select/reject decision.
Use competitor patterns for discovery, never as proof. Do not tag-stuff or attach fashionable AI/privacy/open-
source terms without current product evidence.

## Platforms and licensing

Platform evidence rules:

- Online/SaaS for browser-delivered apps
- Mac/Windows/Linux only for real desktop apps
- Android/iPhone/iPad only for real native/mobile-store apps
- Self-Hosted only for a user-deployable server distribution
- browser extension platforms only for a real supported extension
- Custom Platform only when the product runs within/extends that platform

Licensing rules are in `eligibility-and-submission.md`. Record cost and source license separately. Free,
Freemium, Commercial, Proprietary, and Open Source describe different dimensions; mirror the live controls.

## Alternative association test

For each candidate:

| Test | PASS standard |
|---|---|
| Same main task | Both products solve the same primary job |
| Same focus | Users know both primarily for that job |
| Search intent | A person seeking alternatives to X would reasonably consider this |
| Functional sufficiency | Product can replace the core workflow, not only one feature |
| Honest differences | Platform, pricing, hosting, and audience differences are recorded |

Reject associations based only on shared tags, audience, integrations, data exchange, secondary features,
brand competition, or SEO keywords. Prefer five excellent associations to dozens of weak ones.

After the app is public, use `Contribute to this page` → `Suggest Alternatives` on target profiles. Existing
apps can be searched and suggested; an unlisted app may be added as a new alternative. Admins verify actions.

## Profile media and quality score

Use a crisp square transparent icon. Include screenshots for:

1. the primary job
2. the main output/result
3. setup or admin experience
4. a meaningful differentiator
5. supported platforms or collaboration where useful

Quality score:

| Dimension | Passing standard |
|---|---|
| Truth | Every claim/taxonomy value is current and evidenced |
| Task clarity | Main purpose is clear in one sentence |
| Alternative relevance | Associations pass main-task-and-focus test |
| Taxonomy | Type, tags, and features are useful rather than stuffed |
| Platform accuracy | Only real delivery platforms are selected |
| License accuracy | Free/trial/source status matches reality |
| Media | Required icon and current screenshots meet live contract |
| Moderation usefulness | Optional note explains ambiguous choices concisely |
| Maintainability | Sources, owners, and verification dates are recorded |
